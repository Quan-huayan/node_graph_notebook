# 第一性原理批判：docs/use_cases

## 一、根本问题：这套文档解决的是什么本质问题？

一个"节点图谱笔记本"的本质需求是：**用户创建知识片段（节点），建立片段间的关系（连接），并以可视化方式浏览和检索这些知识**。从这个本质出发，所有设计决策应当被追问：**这是否是从本质需求推导出的最小必要复杂度？**

答案：**不是。** 以下逐层批判。

---

## 二、架构过度工程化

### 2.1 CQRS 模式的正当性存疑

CQRS 的核心价值在于：读写负载不对称的分布式系统中，读写模型可以独立伸缩和优化。但本项目是**单用户、本地文件系统的桌面应用**。

- `README.md:9` 声明采用 CQRS 模式，但未给出任何理由说明为什么一个本地应用需要 Command/Query 分离。
- 实际后果：每个操作都需要定义 Command 类 + Handler 类 + Service 方法 + Repository 方法，四层间接。例如创建一个节点要经过 `CreateNodeCommand → CreateNodeHandler → NodeService.createNode() → NodeRepository.save()`，而本质操作只是"把一个 JSON 写入文件"。
- 更严重的是，`02_graph_management.md:127-130` 中"加载图"使用了 `LoadGraphCommand`——**加载是读操作，却走了 Command 通道**，这直接违反了 CQRS 的写用 Command、读用 Query 的基本原则，说明 CQRS 在此项目中并未被真正理解和一致执行。

### 2.2 中间件管道的必要性

`README.md:91-96` 列出四层中间件：Logging → Transaction → Validation → Undo。

- **TransactionMiddleware**：对本地文件系统做"事务管理"——文件系统本身不支持事务。如果写入 `nodes/{nodeId}.json` 失败，"回滚"意味着什么？删除已写入的文件？如果删除也失败呢？这个中间件给出的是事务的**幻觉**，而非事务的**保证**。
- **LoggingMiddleware** 对每个操作记录时间戳和耗时——对调试有用，但作为核心中间件而非可选插件，违反了项目自身声称的"插件化架构"理念。
- **UndoMiddleware** 被列为核心中间件，但**全部 7 个用例文档中没有任何一个用例描述撤销/重做流程**。一个核心功能没有对应的用例，说明设计时并未真正思考其实现。

### 2.3 QueryCache 的合理性

`07_search_query.md:86-90` 描述了查询缓存：最大 1000 条目，5 分钟 TTL，LRU 淘汰。

- 对于一个本地桌面应用，节点数据量通常在百到千级别，全部加载到内存也不过几 MB。缓存解决的是"重复计算"问题，但如果数据已在内存中，搜索本身就是 O(n) 遍历——缓存搜索结果比直接遍历更快吗？
- 更根本的问题：缓存引入了**一致性问题**。`07_search_query.md:409-421` 描述了缓存失效策略，但"NodeUpdatedEvent → 使相关查询缓存失效"中的"相关"如何定义？一次更新可能影响无数查询的缓存——精确失效的复杂度可能超过缓存本身带来的收益。

## 三、数据模型存在根本性矛盾

### 3.1 连接的归属问题：双重存储

`01_node_management.md:196-203` 描述连接存储在节点上：
```
更新源节点的 outgoingConnections
更新目标节点的 incomingConnections
```

`02_graph_management.md:443` 又描述 Graph 有自己的连接列表：
```
List<GraphConnection> connections;
```

**同一个连接关系存在于三个位置**：源节点的出边、目标节点的入边、图的连接列表。这是三重冗余，任何写操作都必须同时更新三处——如果中间任何一步失败，数据就不一致。从第一性原理出发，连接是一个独立实体，应当有单一存储位置，其他位置通过查询派生。

### 3.2 邻接表：派生数据 vs. 独立状态

`02_graph_management.md:152-166` 描述加载图时从节点连接关系构建邻接表，但 `01_node_management.md:270` 和 `02_graph_management.md:201` 又描述连接/断开操作时"更新 AdjacencyList"。

**邻接表到底是节点连接关系的派生视图，还是独立维护的状态？** 如果是派生视图，就不应该被"更新"，而应该在源数据变化时重建；如果是独立状态，就可能和源数据不一致。文档同时采用了两种矛盾的说法。

### 3.3 节点与图的从属关系模糊

节点独立存储为 `nodes/{nodeId}.json`，图通过 `nodeIds` 列表引用节点。这意味着：
- 一个节点可以属于多个图——但文档没有描述这种多从属关系的语义（删除节点时，所有引用它的图怎么办？）
- 从图中"移除节点"（UC-GRAPH-07）不删除节点本身——那孤立节点如何被发现和清理？
- 删除节点（UC-NODE-03）级联删除连接——但如果节点属于多个图，其他图中的连接是否也被删除？

这些根本性的数据一致性问题在文档中完全没有被讨论。

## 四、AI 集成：从"辅助"到"危险"的鸿沟

### 4.1 Function Calling 缺乏确认机制

`03_ai_integration.md:297-402` 描述 AI 可以通过 Function Calling 直接执行 `create_node`、`delete_node`、`connect_nodes` 等操作。

**AI 可以不经用户确认直接删除节点。** 从第一性原理出发，AI 是一个不可靠的外部服务——它会幻觉、会误解、会执行非预期的操作。允许不可靠代理直接修改用户数据，且无确认步骤，是严重的安全缺陷。

### 4.2 Prompt 构建过于朴素

`03_ai_integration.md:119-123`：
```
4.1 将上下文节点内容拼接成 contextText
4.2 构建 prompt: "Context:\n{contextText}\n\nQuestion: {q}"
```

- 没有 system prompt 来约束 AI 的行为边界
- 没有对话历史管理——多轮对话如何维护上下文？
- 没有 token 限制管理——如果用户选择了 50 个节点作为上下文，直接拼接可能超出模型的上下文窗口
- 连接推荐（UC-AI-05）使用的是简单的字符串相似度匹配（`03_ai_integration.md:265-269`），根本不需要 AI——一个本地字符串匹配函数就够了。如果用 AI 做连接推荐，应该利用语义理解能力，而不是退化为字符串匹配。

### 4.3 API 密钥"加密"存储

`03_ai_integration.md:79` 声称"API密钥加密存储"在 SharedPreferences 中。SharedPreferences 在大多数平台上是**明文存储**的（Android 的 XML、iOS 的 NSUserDefaults、桌面端的 JSON 文件）。所谓"加密"的密钥本身存储在哪里？如果密钥硬编码在应用中，那"加密"毫无意义。

## 五、Lua 脚本：安全模型自相矛盾

### 5.1 "宽松"沙箱 = 没有沙箱

`06_lua_automation.md:480-488`：

| 配置 | 宽松模式 | 严格模式 |
|------|---------|---------|
| 文件IO | 允许 | 禁止 |
| OS调用 | 允许 | 禁止 |
| 网络 | 允许 | 禁止 |
| 执行时间 | 无限制 | 30秒 |
| 内存 | 无限制 | 64MB |

宽松模式下，Lua 脚本可以：执行任意系统命令（`os.execute`）、读写任意文件、访问网络、无限执行时间、无限内存。这等于**没有沙箱**。一个从网上下载的 Lua 脚本可以在宽松模式下完全控制用户的计算机。

更根本的问题：**谁来决定使用哪种模式？** 如果是用户，大多数用户不理解安全风险；如果是脚本作者，恶意脚本自然会选择宽松模式。文档没有描述任何权限模型。

### 5.2 动态 Hook 的无限递归风险

`06_lua_automation.md:317-363` 描述 Lua 脚本可以注册动态 Hook，Hook 回调中可以调用 `app.node.create()` 等操作，这些操作会触发 Command，Command 会发布事件，事件可能触发其他 Hook……**没有描述任何递归深度限制或事件循环检测机制。**

## 六、导入导出：信息丢失的必然性

### 6.1 Markdown 解析规则过于简陋

`05_import_export.md:100-107`：
```
# 标题 → Node.title
内容 → Node.content
--- 分隔符 → 新节点
```yaml 块 → Node.metadata
```

- 没有处理 `##`、`###` 等子标题——它们被当作"内容"吗？
- 没有处理列表、代码块、引用等 Markdown 结构
- 没有处理图片和附件——知识笔记中图片是常见内容
- **没有往返保证**：导入 Markdown → 创建节点 → 导出 Markdown，结果可能和原始文件不同。文档没有承认这一点。

### 6.2 节点位置信息的丢失

导出为 Markdown 时，节点的位置（position）、大小（size）、连接关系（connections）等图结构信息**无法在 Markdown 中表达**。导入后再导出，这些信息就丢失了。文档对此只字未提。

## 七、缺失的根本性用例

从"节点图谱笔记本"的本质需求出发，以下用例是**必然需要的**，但文档中完全缺失：

| 缺失用例 | 为什么是根本性的 |
|----------|----------------|
| **撤销/重做** | UndoMiddleware 是核心中间件，但没有用例描述其流程 |
| **数据完整性检查** | 节点引用图、图引用节点、连接引用两端节点——任何一方的删除都可能导致悬空引用 |
| **冲突解决** | AI 和用户可能同时修改同一节点；Lua 脚本和用户操作可能并发 |
| **数据迁移/格式版本** | 文件格式随版本演进，旧数据如何兼容？ |
| **大图性能** | 文档假设所有节点都能加载到内存，没有分页或懒加载策略 |
| **孤立节点管理** | 不属于任何图的节点如何被发现和清理？ |
| **多选操作** | 用户选择多个节点进行批量操作，但只有 UC-NODE-07 简要提及 |

## 八、文档自身的方法论问题

### 8.1 用例描述的是"实现"而非"需求"

第一性原理要求从需求出发推导设计。但这些文档描述的是**已经做出的架构决策的调用链**，而非用户需求。例如 UC-NODE-01 描述的是"用户点击按钮 → CreateNodeCommand → CommandBus → Handler → Service → Repository"这条实现路径，而不是"用户需要创建一个节点"这个需求及其约束。

这导致文档**无法被用来验证设计的合理性**——因为需求和设计被混为一谈，你无法用需求来检验设计是否过度或不足。

### 8.2 角色定义缺乏实际意义

每个文档都定义了"用户角色"（如"普通用户"、"高级用户"、"AI用户"、"开发者"），但这些角色在用例描述中**从未被引用**。没有任何用例标注"此用例仅对某角色可用"或"不同角色有不同的行为路径"。角色定义成了装饰品。

### 8.3 优先级标注缺乏标准

用例优先级标注为 P0/P1/P2，但没有定义优先级的标准。P0 是"必须有"还是"最先实现"？P2 是"可以不做"还是"延后实现"？没有标准，优先级就没有决策价值。

---

## 九、总结：根本性的设计偏差

从第一性原理审视，这套用例文档的核心问题是：**它从一个简单的本质需求（管理知识节点及其关系）出发，却引入了一个面向企业级分布式系统的架构复杂度**。

| 本质需求 | 文档中的设计 | 差距 |
|---------|------------|------|
| 存储和检索知识片段 | CQRS + 文件系统 + 缓存 + 物化视图 | 一个内存索引 + 定期持久化即可 |
| 建立片段间关系 | 三重冗余存储（节点出边/入边 + 图连接列表 + 邻接表） | 连接是独立实体，单一存储 + 派生视图 |
| 可视化浏览 | Flame 引擎 + LOD + 视锥裁剪 + 空间索引 | 对千级节点，简单的 Canvas 渲染即可 |
| AI 辅助 | Function Calling + 无确认直接执行 | AI 应建议，用户确认后执行 |
| 可扩展性 | 完整插件系统 + 沙箱 + Hook + API Registry | 对 MVP 来说，内置功能模块 + 简单扩展点即可 |

**最关键的批判**：这套文档更像是架构展示文档，而非需求驱动的用例文档。它回答了"系统怎么实现"，但没有回答"为什么需要这样实现"，更没有回答"有没有更简单的方式实现同样的需求"。
